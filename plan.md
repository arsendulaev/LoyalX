# План разработки программного комплекса «LoyalX»

### **1. Общее описание проекта**

*   **Название:** LoyalX
*   **Концепция:** Кросс-продуктовая он-чейн система лояльности.
*   **Проблема:** Фрагментация программ лояльности (много карт/приложений) и низкая ликвидность накопленных баллов (их сложно потратить).
*   **Решение:** Децентрализованная платформа на базе блокчейна TON, интегрированная в Telegram (Mini App), где баллы являются криптографическими токенами (Jettons). Пользователи хранят баллы в личном кошельке и могут обменивать их между брендами-партнерами.
*   **Целевая аудитория:**
    *   **Пользователи:** Клиенты ритейла и сферы услуг.
    *   **Бизнес:** Малые и средние предприятия (кафе, магазины, барбершопы).

### **2. Архитектура системы**

Система состоит из трех ключевых компонентов:

1.  **On-chain (Блокчейн-часть):**
    *   *Назначение:* Хранение критически важных данных (балансы, правила) и выполнение бизнес-логики (эмиссия, обмен).
    *   *Состав:* Набор смарт-контрактов, написанных на языке **Tact**.
        *   **`Factory.tact`**: Главный контракт-«фабрика», отвечающий за регистрацию новых брендов и выпуск для них собственных контрактов-жетонов.
        *   **`BrandJetton.tact`**: Смарт-контракт жетона (балла лояльности), соответствующий стандарту TEP-74. Содержит логику прямого обмена (Swap) на жетоны других брендов по заданному курсу.

2.  **Frontend (Клиентская часть):**
    *   *Назначение:* Пользовательский интерфейс для взаимодействия с системой.
    *   *Реализация:* Telegram Mini App (TMA), построенное на **React + TypeScript**.
    *   *Функции:* Подключение криптокошелька, отображение балансов, инициирование транзакций (создание бренда, обмен), сканирование QR-кодов.

3.  **Off-chain (Вспомогательная часть):**
    *   *Назначение:* Хранение "тяжелых" и некритичных данных (метаданные токенов).
    *   *Реализация:* Статический хостинг (например, **GitHub Pages**) для JSON-файлов и изображений (логотипов). Смарт-контракты будут хранить только ссылки (URL) на эти файлы.

### **3. Технологический стек**

| Компонент | Технология | Назначение |
| :--- | :--- | :--- |
| **Блокчейн** | TON (The Open Network) | Основная среда выполнения. |
| **Смарт-контракты** | Tact | Язык программирования для логики. |
| **Среда разработки** | Blueprint | Фреймворк для компиляции, тестирования и деплоя контрактов. |
| **Frontend** | React, Vite, TypeScript | Создание пользовательского интерфейса. |
| **Интеграция с TON** | `@tonconnect/ui-react`, `ton-core` | Подключение кошельков и взаимодействие с блокчейном. |
| **Хранение кода** | Git, GitHub | Управление версиями, хостинг метаданных. |
| **IDE** | Visual Studio Code | Основная среда разработки. |

### **4. Этапы разработки и задачи (MVP)**

##### **Этап 0: Подготовка инфраструктуры**
*   **Задача 0.1:** Создать монорепозиторий на GitHub.
*   **Задача 0.2:** Настроить структуру папок: `contracts/` и `frontend/`.
*   **Задача 0.3:** Инициализировать проект Blueprint в папке `contracts`.
*   **Задача 0.4:** Инициализировать проект Vite (React+TS) в папке `frontend`.

##### **Этап 1: Разработка On-chain логики (Смарт-контракты)**
*   **Задача 1.1: Создание контракта `BrandJetton.tact`**
    *   *Подзадача 1.1.1:* Реализовать базовый стандарт TEP-74 (сообщения `Transfer`, `Burn`).
    *   *Подзадача 1.1.2:* Добавить логику хранения метаданных (ссылка на off-chain JSON).
    *   *Подзадача 1.1.3:* Реализовать функцию приема жетонов (`receive_jettons`) для логики обмена (Swap): контракт проверяет, от кого пришли жетоны, смотрит курс в своей таблице и отправляет в ответ свои жетоны.
    *   *Подзадача 1.1.4:* Написать unit-тесты (на TypeScript) для проверки эмиссии (mint), сжигания (burn) и обмена (swap).

*   **Задача 1.2: Создание контракта `Factory.tact`**
    *   *Подзадача 1.2.1:* Реализовать хранение счетчика брендов (`nextBrandId`).
    *   *Подзадача 1.2.2:* Написать `receive`-функцию, которая принимает сообщение `CreateBrand`.
    *   *Подзадача 1.2.3:* Реализовать логику деплоя нового экземпляра контракта `BrandJetton` с уникальным адресом.
    *   *Подзадача 1.2.4:* Написать unit-тесты, проверяющие, что фабрика корректно создает новые контракты и увеличивает счетчик.

##### **Этап 2: Разработка Frontend (Telegram Mini App)**
*   **Задача 2.1: Базовая настройка и авторизация**
    *   *Подзадача 2.1.1:* Настроить основной layout приложения.
    *   *Подзадача 2.1.2:* Интегрировать кнопку "Connect Wallet" с помощью `@tonconnect/ui-react`.

*   **Задача 2.2: Экран "Кошелек"**
    *   *Подзадача 2.2.1:* Сверстать интерфейс для отображения списка токенов лояльности.
    *   *Подзадача 2.2.2:* Реализовать логику чтения балансов жетонов с подключенного кошелька.

*   **Задача 2.3: Экран "Создать Бренд"**
    *   *Подзадача 2.3.1:* Сверстать форму для ввода данных (название, тикер).
    *   *Подзадача 2.3.2:* Реализовать функцию, которая формирует и отправляет транзакцию `CreateBrand` на контракт `Factory`.

*   **Задача 2.4: Экран "Обмен" (Swap)**
    *   *Подзадача 2.4.1:* Сверстать интерфейс с полями "Отдаю" / "Получаю".
    *   *Подзадача 2.4.2:* Реализовать логику отправки жетонов на контракт другого бренда для инициации обмена.

##### **Этап 3: Интеграция и тестирование**
*   **Задача 3.1:** Написать скрипты деплоя (в `scripts/`) для развертывания `Factory` в **Testnet**.
*   **Задача 3.2:** "Прошить" в код фронтенда адрес развернутого контракта `Factory`.
*   **Задача 3.3:** Провести полное E2E (End-to-End) тестирование:
    1.  Зайти в приложение -> Подключить кошелек.
    2.  Создать "Бренд А" через интерфейс.
    3.  Создать "Бренд Б" через интерфейс.
    4.  Начислить себе баллы "Бренда А".
    5.  Обменять баллы "Бренда А" на баллы "Бренда Б".

##### **Этап 4: Финализация и развертывание**
*   **Задача 4.1:** Разместить файлы метаданных (JSON, PNG) на GitHub Pages.
*   **Задача 4.2:** Развернуть фронтенд на статическом хостинге (Vercel, Netlify, GitHub Pages).
*   **Задача 4.3:** Создать Telegram-бота в `@BotFather` и привязать к нему URL развернутого приложения.

### **5. Git Workflow**

*   **`main`**: Стабильная, рабочая ветка. Слияние в `main` только через Pull Request.
*   **`feat/name-of-feature`**: Ветки для разработки нового функционала (например, `feat/swap-screen`).
*   **Коммиты**: Использовать семантические сообщения (`feat:`, `fix:`, `docs:`, `test:`).

### **6. Определение MVP (Минимально жизнеспособный продукт)**

Для успешной защиты курсового проекта MVP должен демонстрировать следующий функционал:
1.  Пользователь может подключить свой TON-кошелек к приложению.
2.  Пользователь (в роли бизнеса) может создать новую программу лояльности (токен) через интерфейс.
3.  Пользователь может видеть баланс созданных токенов.
4.  Пользователь может успешно обменять токены одного бренда на токены другого.
